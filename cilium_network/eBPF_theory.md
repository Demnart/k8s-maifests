# eBPF теория

## Что такое eBPF  

eBPF(extended Berkeley Packet Filter) - технология ядра Linux, позволяющая запускать пользовательские программы непосредственно в ядро без необходимости изменять исходный код ядра или загружать модули.  

## Краткая история  

Технология берёт начало от классического BPF( Berkeley Packet Filter), созданного в 1992 году Стивено МакКэнном и Ван Джейкобсоном. Оригинальный BPF использовался для эффективной фильтрации сетевых пакетов - например, в утилите ```tcpdump```.  

В 2014 году Алексей Старовойтов кардинально расширил BPF для Linux. Оригинальная версия была переименованна в cBPF(classic BPF), а новая получила название eBPF. Начиная с ядра 3.18(декабрь 2014), eBPF включён в Linux как полноценная подсистема.  

Сегодня eBPF - это уже не просто фильтрация пакетов. Технология применяется для сетевой обработки, наблюдаемости, безопасности и трейсинга. В 2021 году создан eBPF Foundation при Linux Foundation.  

## Как работает eBPF  

Процесс исходного кода до выполнения: 

```sh
Исходный код (С)
    |
    v
Компиляция в ядро (системный вызов bpf())
    |
    v
Верификация (проверка безопасности)
    |
    v
JIT-компиляция (байткод -> нативные инстркуции)
    |
    v
Прикрепление к хуку (точке перехвата в ядре)
```

1. Программа пишется на ограниченном подмножестве C.  
2. Clang/LLVM компилирует её в eBPF байткод.  
3. Загрузчик вызывает системный вызов bpf() для передачи байткода в ядро.  
4. ```Верификатор``` проверяет безопасность программы(подробнее ниже).  
5. ```JIT-компилятор``` транслирует байткод в нативные машинные инструкции - после этого программа работает с эффективностью нативного кода ядра.  
6. Программа присоединяется к определённой точке (хуку) и вызвается при наступлении события.  

## Ключевые компоненты  

```Хуки (hook points)``` - точки в ядре, к которым прикперепляются eBPF-программы: 

| Хук | Где работает | Назначение |
| :--- | :--- | :--- |
| XDP | В драйвере сетевого устройства, до сетевого стека | Самя рання точка обработки пакетов. Максимальная производительность |
| TC | После начальной обработки до L3 | Ingress/egress обработка с доступом к метаданным пакета |
| Socket | На уровне cgroup/сокета| Перехват connect(), sendmsg() и других операций|
| Kprobes | Вход/выход из функции | Трейсинг и отладка|
| Tracepoints | Статические точки трассировки | Наблюдаемость |  

```BPF Maps``` - структуры данных, разделяемые между eBPF-программами в ядре и пользовательском пространстве:  

| Тип | Описание |
| :--- | :--- |
| Hash | Хеш-таблица, O(1) lookup по производному ключу |
| Array | Массив с индексацией по целом числу |
| Per-CPU Hash/Array | Отдельный экземпляр для каждого CPU( без блокировок)|
| LRU Hash| Хеш-таблица с вытеснением наименее используемых записей |
| LPM Trie | Longest prefix match - для IP-маршрутизации |
| Ring Buffer| Кольцевой буфер для потоковой передачи событий |

**Helper-функции** - набор функций ядра, доступных из eBPF-программ. Примеры:  

  - ```bpf_map_lookup_elem()``` - поиск в map.  
  - ```bpf_redirect()``` - перенаправление пакета.  
  - ```bpf_get_current_pid_tgid()``` - получение PID процессов.  


## Верификатор  

Верфикатор - критически важный компонент безопасности. Это абстрактный интерпритатор, который симулирует все возможные пути выполнения программы перед загрузкой в ядро.  

**Что проверяет**:  

  - Программа **всегда завершается** - нет бесконечных циклов (с ядра 5.3 допускаются ограниченные циклы с доказуемо конечным числом итераций).  
  - Нет чтения **не инициализированной** или **произвольной** памяти ядра.  
  - Все обращения к памяти - в пределах допустимых границ.  
  - Программа **не может вызвать panic ядра**.  

**Ограничения**:  

  - Стек: 512 байт на функцию.  
  - Лимит: 1 000 000 проверенных инструкций.  
  - Ограниченное число вложенных вызовов.  

Благодаря верификатору загрузка eBPF-программы в ядро безопасна - некорректная программа будет отклонена до начала выполнения.  

## eBPF в сетевой подсистеме Kubernetes  

### Традиционный подход: iptables/IPVS  

В стандартной конфигурации Kubernetes за обработку сервисов отвечает **kube-proxy**: 

```sh
Под -> iptables/IPVS правила -> DNAT -> conntrack -> Целевой под
```

**kube-proxy с iptables**:  

  - Для каждого Service и Endpoint создается набор правил iptables.  
  - При росте кластера количество правил растет линейно - O(n) при каждом обходе цепочки.  
  - Обновление правил - полная перезапись цепочки.  
  - NAT/DNAT выполняются через conntrack - дополнительная нагрузка.  

**kube-proxy c IPVS**:  

  - Использует хеш-таблицы для балансировки(лучше маштабируется).  
  - Но по-прежнему зависит от iptables для SNAT и маскарадинга.  
  - Всё ещё требует conntrack.  

### Подход eBPF  

eBPF-программы обрбатывают пакеты непосредственно в ядре, минуя iptables:  

```sh
Под -> eBPF (socket/TC/XDP) -> BPF map lookup O(1) -> Целевой под
```  

Ключевые отличия:  

  - **O(1) lookup** - сервисы и endpoints хранятся в BPF hash maps. Поиск не зависит от количества сервисов в кластере.  
  - **Socket-level балансировка** - перехват на уровне ```connect()``` позволяет подменить адрес назначения **до** создания пакета. Нет per-packet NAT и conntrack.  
  - **XDP** - для входящего трафика обработка на самом раннем этапе( в драйвере сетевого устройства), до прохождения сетевого стека.  

## Что делает Cilium с eBPF  

[Cilium](https://cilium.io/) - Graduated-проект CNCF, использующией eBPF для реализации сетевой подсистемы Kubernetes:  

  - **Pod networking** - eBPF-программы на TC-хуках veth-интерфейсов обеспечивают связность подов. Пакеты между подами на одной ноде перенаправляются напрямую через ```bpf_redirect```.  
  - **Балансировка сервисов** - полная замена kube-proxy. Внутрикластерный трафик обрабатывается на уровне сокетов (без NAT). Воходящий - на XDP.  
  - **Network Policies** - Cilium использует числовые Security Identity вместо IP-адресов. Идентификатор формируется из labels пода и сохраняется при пересоздании пода с новым IP. Поддержка L3/L4 и L7(HTTP, gRPC, Kafka).  
  - **Наблюдаемость** - Hubble собирает flow-данные прямо из eBPF-хуков без дополнительного overhead.  

## Преимущества и ограничения  

**Преимущества**:  

  - **Производительность** - О(1) lookup сервисов вместо O(n) iptables-цепочек. Socket-level балансировка устраняет per-packet NAT. JIT-компиляция обеспечивает производительность на уровне нативного кода.  
  - **Программируемость** - загрузка и обновление логики без перекомпиляции ядра.  
  - **Наблюдаемость** - глубокая видимость сетевых потоко с минимальным overhead.  
  - **Маштабируемость** - производительность не деградирует при росте числа сервисов.  

**Ограничения**:  

  - **Требования к ядру** - Cilium 1.19 требует Linux > 5.10. Для полной функциональности рекомендуется 5.10+.  
  - **Сложность отладки** - eBPF-программы работают в ядре, стандартные инструменты отладки не применемы. Верификтор может отклонять программы по неочевидным причинам.  
  - **Потребление памяти** - BPF maps занимают pinned kernel memory(не выгружаются в swap). При большом количестве сервисов и endpoints расход памяти может быть значительным.  
  - **Кривая обучения** - eBPF - принципиально иная парадигма по сразвнению с iptables. Для глубокой диагностики нужно понимание работы ядра и инструментов экосистемы(bpftool, bpftrace).
